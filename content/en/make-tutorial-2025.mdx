<h2>2) Core concepts that matter</h2>
<h3>Visual scenarios &amp; mapping</h3>
<p>
          Build automations as <em>scenarios</em> on a canvas. Each module’s inputs and outputs become fields in the
          <strong>mapping</strong> panel, where you drag data, transform values (functions), and navigate arrays/collections.
          You can run a single module to “learn” its output structure, then map confidently to the next steps.
        </p>
<h3>Routers, iterators &amp; aggregators</h3>
<p>
<strong>Routers</strong> branch your flow into multiple routes with filters and a possible <em>fallback</em> route (processed sequentially).
          <strong>Iterators</strong> split an array into one bundle per element; use an <em>array aggregator</em> afterwards if you need to re-combine results.
        </p>
<h3>Triggers: polling vs webhooks</h3>
<p>
          Polling modules check for changes on schedule. <strong>Webhooks</strong> receive HTTP requests and typically trigger scenarios instantly (parallel by default).
          If you prefer ordered processing or have shared resources (e.g., a rate-limited API), enable <em>sequential</em> execution.
        </p>
<h3>Error handling &amp; incomplete executions</h3>
<p>
          Attach error handlers (ignore, resume, commit, rollback, break) to handle expected faults without disabling your scenario.
          Enable <em>incomplete executions</em> to store a failed run’s state for later retry and to avoid data loss.
          For transient issues (rate limits, timeouts), Make applies <em>exponential backoff</em> for automated re-runs.
        </p>
<h3>Data Stores &amp; Data Structures</h3>
<p>
<strong>Data Stores</strong> are Make’s built-in, lightweight database for key business data (dedupe sets, cursors, id caches).
          <strong>Data Structures</strong> formally describe JSON/XML/CSV schemas so mapping is predictable and validated.
        </p>
<h3>AI Agents &amp; context</h3>
<p>
          Agents can use your scenarios as tools, keep <em>context</em> (files/text) as long-term memory, and run within timeout/step limits you configure.
          You can either use Make’s AI provider (dynamic credit usage) or your own AI connection on Pro+ plans.
        </p>

<h2>3) Step-by-step tutorial (with diagrams)</h2>
<h3>A) Prepare your workspace</h3>
<ol>
<li>Create a team and folders (e.g., <em>E-commerce</em>, <em>Content</em>, <em>Ops</em>). Name scenarios clearly (prefix by domain and intent).</li>
<li>Set environment variables (API keys) as <em>Connections</em>. For sensitive flows, use a dedicated team with limited members.</li>
<li>Plan <em>inputs/outputs</em> for scenarios that are used as tools (Agents or other scenarios). Use “Scenario inputs/outputs” to define types and validation.</li>
</ol>

<h3>B) Build a real-time lead intake (Webhook -&gt; CRM)</h3>
<ol>
<li><strong>Trigger:</strong> Webhooks &gt; <em>Custom webhook</em>. Copy the URL; your form/landing page will POST here.</li>
<li><strong>Validation:</strong> Add a <em>JSON &gt; Parse JSON</em> with a <em>Data Structure</em> (define required fields). If invalid, route to an error handler that alerts Slack.</li>
<li><strong>Routing:</strong> Add a <em>Router</em> for VIP vs standard leads (filters on UTM/source or spend). Mark a <em>fallback route</em>.</li>
<li><strong>CRM upsert:</strong> Use native CRM module (or HTTP) to <em>find-&gt;create/update</em>. Consider <em>sequential</em> scenario if the CRM API enforces strict rate limits.</li>
<li><strong>Notify:</strong> Slack or email summary (mapped fields). Include a link back to your CRM record.</li>
<li><strong>Observability:</strong> Enable <em>incomplete executions</em> and create a lightweight <em>Data Store</em> for dedupe keys (email+source).</li>
</ol>
<h3>C) Order pipeline (Shopify -&gt; Notion/Airtable + Gmail/Slack)</h3>
<ol>
<li><strong>Trigger:</strong> Shopify order creation (instant, if available) or polling.</li>
<li><strong>Iterator:</strong> Iterate line items. For each item, normalize SKUs and collect totals.</li>
<li><strong>Branching:</strong> Router by fulfillment rule (dropship vs in-house) and customer tier.</li>
<li><strong>Write:</strong> Append a Notion/Airtable row with mapped fields; attach invoice PDF if present.</li>
<li><strong>Comms:</strong> Send a personalized Gmail message; post a concise Slack summary with order link.</li>
<li><strong>Resilience:</strong> Error handler -&gt; rollback Data Store updates or flag the order for manual review (incomplete execution).</li>
</ol>
<h3>D) Editorial calendar (AI-assisted)</h3>
<ol>
<li>Trigger from a Notion/Airtable “ideas” table or via webhook from your CMS.</li>
<li>Use Make AI Toolkit or your preferred AI app to <em>classify</em> ideas and generate outlines (credits: fixed + token-based if using Make’s provider).</li>
<li>Create tasks, draft documents, and due dates across Apps (Notion/Trello/Asana). Keep <em>Data Store</em> cursors to avoid duplicate drafts.</li>

<div className="stack">
<h2>5) Reliability: errors, retries, rate limits</h2>
<h3>Webhooks &amp; throughput</h3>
<p>
          Instant webhooks execute immediately and runs are <em>parallel</em> by default. If strict ordering matters (payments, stock),
          toggle <strong>sequential processing</strong>. Make can handle up to <strong>30 incoming webhook requests/second</strong> per webhook; above that you’ll get HTTP 429.
        </p>
<h3>Default exponential backoff</h3>
<p>
          For transient errors (<code>RateLimitError</code>, <code>ConnectionError</code>, <code>ModuleTimeoutError</code>), Make automatically schedules re-runs with
          increasing delays. With incomplete executions enabled, the retry ladder goes ~1 min -&gt; 10 min -&gt; ... up to hours before disabling the schedule if failures persist.
        </p>
<h3>Error handlers (5 types)</h3>
<ul>
<li><strong>Ignore</strong> (drop/continue), <strong>Resume</strong> (continue with defaults), <strong>Commit</strong> / <strong>Rollback</strong> (for ACID-capable modules), <strong>Break</strong> (force incomplete execution).</li>
</ul>
<h3>Practical pattern</h3>
<ol>
<li>Enable <em>incomplete executions</em> and <em>sequential</em> when writing to the same record to avoid race conditions.</li>
<li>Use a Data Store entry to guard idempotency (e.g., hash of payload).</li>
<li>Put <em>Webhook response</em> near the end if you rely on “error disables scenario” behavior for visibility.</li>
</ol>
</div>

<div className="stack">
<h2>6) Data Stores, Data Structures &amp; mapping</h2>
<h3>Data Stores</h3>
<p>
          Think of Data Stores as quick, structured tables inside Make:
          perfect for dedupe, incremental cursors, small reference dictionaries, or temporary queues. They are not a replacement for your main DB, but
          they eliminate a lot of glue code for small stateful needs.
        </p>
<h3>Data Structures</h3>
<p>
          Define and reuse schemas for JSON/XML/CSV. Structures validate inputs/outputs and make mapping predictable.
          Use the generator by pasting a sample payload, then tweak types and required fields.
        </p>
<h3>Mapping arrays</h3>
<p>
          You can index array elements in the mapping UI and combine array functions to pick values by key.
          When you lack structure info (e.g., after Parse JSON), run the module once so Make “learns” its output and exposes fields downstream.
        </p>
</div>

<ul>
<li>Collect consent on your site; fire GA4 only after Analytics is accepted. Use Consent Mode v2 and store choices with a timestamp.</li>
<li>Log sensitive actions (create/update customer, refunds) with minimal data in a secure system; avoid putting secrets in scenario notes.</li>

<section className="section" id="limits">
<div className="stack">
<h2>10) Practical limits &amp; when not to automate</h2>
<ul>
<li><strong>Unstable APIs:</strong> use retries/backoff and alerts, but accept manual fallbacks when providers are flaky.</li>
<li><strong>High-risk sequences:</strong> keep destructive actions (<em>delete</em>, <em>refund</em>) behind explicit filters &amp; approvals.</li>
<li><strong>Mega-scenarios:</strong> split into smaller on-demand tools chained via webhooks; this scales better and debugs faster.</li>
<li><strong>X/Twitter workflows:</strong> plan alternatives (Buffer, Hootsuite, Mastodon, Bluesky) since the X app was discontinued in 2025.</li>
</ul>
